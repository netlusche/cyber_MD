# CyberMD Test Automation Guide

This document outlines the testing infrastructure, how to execute tests locally, and the specific scenarios covered to prevent regressions in the *CyberMD* application.

## 🚀 Running the Tests Locally

We use a combination of **Vitest** for fast, component-level testing and **Playwright** for complete End-to-End (E2E) browser interactions.

### Component & Unit Tests (Vitest)
These tests validate internal logic, component state (Zustand), DOM rendering, and React state independent of complex browser APIs.

```bash
# Run all Vitest tests once
npm run test

# Run tests in watch mode (ideal during development)
npx vitest

# Run tests with UI interface
npx vitest --ui
```

### End-to-End Tests (Playwright)
Playwright spins up real browser instances to test how the complete application works, including file downloads, rendering layout shifts, and viewport responsiveness.
*Note: Make sure your Vite dev server (`npm run dev`) is not running on port 5173 to avoid conflicts, or Playwright will reuse it.*

```bash
# Run all E2E tests in headless mode (Chromium, Webkit, Firefox, Mobile)
npx playwright test

# Run tests specifically in Chromium
npx playwright test --project chromium

# Run tests in UI mode for debugging
npx playwright test --ui
```

## 🧪 Implemented Test Cases & Scenarios

To ensure application stability, we've implemented both a Baseline functionality suite and a targeted Regression suite based on historical bugs from Release 1.2.

### Baseline Functional Suite 
*(Located in `src/test/core.test.tsx`)*

These tests ensure the standard Markdown editing experience remains functional:
1. **Markdown Compilation & Auto-Save:** Verifies that typing into the Tiptap editor successfully parses Markdown, updates the Zustand store (`html`/`markdown`), and persists to `localStorage`.
2. **Formatting Tools & Complex Plugins:** Mounts the application and verifies the DOM presence of all formatting options (Bold, Italic, Images, Tables) to ensure no toolbar regressions.
3. **Layout Modes (WYSIWYG, SPLIT, CODE):** Dynamically switches between layout modes and validates that the correct container styles (`display: none` or flex structures) are applied.
4. **Focus Modes (ZEN, TYPEWRITER):** Activates specific focus modes and asserts that the `app-header` properly unmounts or centers the content.
5. **File I/O Mock Testing:** Simulates launching the native File System Access API dialog (`showOpenFilePicker`) using mocks, ensuring the internal `cybermd-command` event structure is sound.
6. **Theming Engine:** Cycles through themes via the dropdown and verifies that the `data-theme` attribute correctly updates on the document root to inject CSS variables.

### Regression Suite (Preventing Past Bugs)
*(Located across `regression.test.tsx` and `regression.spec.ts`)*

1. **PDF Export Reliability ("The 0-Byte PDF Bug" & "IndexSizeError")**
   * **Context:** PDFs generated by `html2pdf.js`/`html2canvas` would frequently render completely empty (0 bytes) strictly due to two major bugs: 
     - **File System Access API (Mac/Safari):** Streaming standard `Blob` objects to `FileSystemWritableFileStream.write()` failed silently internally on Safari, producing empty files. Fixed by converting the stream output explicitly to `ArrayBuffer` logic.
     - **IndexSizeError (text-transform):** The `html2canvas` tree parser crashed dynamically when evaluating CSS `text-transform: uppercase` on strings that expanded mathematically (e.g. the German 'ß' to 'SS'). The DOM was updated with a sanitization step (`sanitizeNode()`) that strips zero-width spaces (`\uFEFF`) and normalizes case strings natively before cloning.
     - **Styling Enforcement:** Headings are now explicitly overridden via an injected `<style>` tag locking generic System UI Sans-Serif fonts, replacing the neon Cyberpunk `Orbitron` fonts on generation.
   * **Test (`regression.spec.ts`):** We use a real Chromium instance via Playwright. The native file picker API is intercepted (`delete window.showSaveFilePicker`), forcing a fallback standard `<a download>` event. We intercept the download exclusively and assert that the file size is robust (`> 180,000 bytes`).

2. **Responsive Toolbar & Flexbox Wrapping ("The Disappearing Menu Bug")**
   * **Context:** The `ResizeObserver` logic caused the advanced "More Tools" (`...`) button to either vanish or wrap onto a new line breaking the UI at strict breakpoints.
   * **Test (`regression.spec.ts`):** Playwright resizes the viewport explicitly to `1024px`, `700px`, and `390px`. It forces WYSIWYG mode to ensure the editor width maps to the viewport perfectly, and verifies that the exact number of expected toolbar buttons are visible.

3. **Editor Unmounting in Split Layouts ("The Dead Action Buttons Bug")**
   * **Context:** Switching from `WYSIWYG` to `CODE` preview completely unmounted the Tiptap `<Editor />`. "NEW" and "LOAD" actions would fail silently because the event listener died with the component.
   * **Test (`regression.test.tsx`):** We switch to CODE layout, ensure the `.tiptap` editor wrapper remains physically in the DOM (`display: none`), and then fire a `cybermd-command` event, asserting the state continues to update.

4. **DOM/CSS Overrides - Tiptap Checkboxes ("The Ugly Browser Style Bug")**
   * **Context:** Default Tiptap checkboxes ruined the Cyberpunk aesthetic.
   * **Test (`regression.test.tsx`):** Renders a task list, counts the injected `input[type="checkbox"]` elements natively handled by the custom CSS in `App.css`.

5. **Plain Light Theme Contrast**
   * **Context:** Buttons disappeared against active backgrounds in the light theme.
   * **Test (`regression.test.tsx`):** Switches the theme selector to `Plain Light` and validates that layout buttons maintain standard DOM class structures like `.btn-active` to hook into our high-contrast CSS mappings.

6. **DOCX Export Semantic Mapping ("The Word Document Integrator")**
   * **Context:** Native export of WYSIWYG models logic into Microsoft Word `.docx`.
   * **Mechanism:** Bypasses `html2canvas` rendering entirely and utilizes `html-docx-js-typescript` to deeply parse the `Editor`'s `useAppStore().html` semantic Tiptap HTML string (`<h1>`, `<ul>`, etc.). Writes exactly through the same File System Access API stream logic established for the PDF engine.
   * **Test Scenario:** Validate that the "EXPORT .DOCX" API triggers correct event dispatches (`cybermd-export-docx`) gracefully from both the primary UI dropdown and the inner `CodeEditorPane` dropdown. Verify generated Microsoft DOCX blob parsing does not break the layout stream hooks.
